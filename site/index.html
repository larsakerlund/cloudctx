One machine. Many customers.
Zero cross-contamination.

cloudctx pins each iTerm2 window to exactly one named cloud identity, so az login or az account set in one window can never silently change the active subscription in another.

The problem

The Azure CLI keeps all of its state — the MSAL token cache and the active subscription — inside a single directory, AZURE_CONFIG_DIR (default ~/.azure). Every shell shares it. Log in to a different customer tenant in one window and every other window silently repoints. If you work with multiple customers, that is an incident waiting to happen.

ctx use acme

How it works

A context is a credential bundle under ~/.cloudctx/<name>/. Selecting one exports a fixed set of environment variables into the current shell — and because the active Azure subscription lives inside AZURE_CONFIG_DIR, two windows with different exports cannot interfere.

VariablePoints at
AZURE_CONFIG_DIR~/.cloudctx/<name>/azure/ — token cache + active subscription
AWS_CONFIG_FILE~/.cloudctx/<name>/aws/config (when the context defines AWS fields)
AWS_SHARED_CREDENTIALS_FILE~/.cloudctx/<name>/aws/credentials (ditto)
AWS_PROFILEthe context's profile name (ditto)

A tiny sourced shell function (ctx) does the part a binary can't — mutating the parent shell — and everything else is a zero-dependency Python CLI.

ctx --help

Features

The guard

Run az or aws with no context selected and the shim warns you before you touch the global default store.

iTerm2 dynamic profiles

Every context becomes an iTerm2 profile: pick a customer from the menu, get a correctly scoped, color-coded window.

Starship segment

The active context sits in your prompt, so you always know which identity a window holds.

ctx exec

Run one command in a context — ctx exec acme -- terraform plan — without changing the shell you're in.

Claude Code ready

Agents inherit the shell environment, so any az or Terraform they run stays inside the window's context.

AWS-ready registry

AWS fields are first-class and exported when present. SSO login automation is on the roadmap.

cloudctx install --write

Install

Requires Python 3.9+ (stdlib only) and, for Azure auth, the Azure CLI.

install
git clone https://github.com/eliknut/cloudctx ~/code/cloudctx
ln -s ~/code/cloudctx/cloudctx ~/bin/cloudctx
cloudctx install --write   # hook the ctx shim into your shell rc

ctx <command>

Usage

usage
ctx new acme --display "Acme AB" --color "#c0392b" \
    --azure-tenant <tenant-guid> --azure-subscription "<sub>"
ctx use acme      # scope THIS shell + badge the window
ctx status        # the REAL identity here: az account show
ctx exec acme -- terraform plan
ctx claude acme ~/proj   # scoped iTerm2 window running Claude Code
ctx clear
