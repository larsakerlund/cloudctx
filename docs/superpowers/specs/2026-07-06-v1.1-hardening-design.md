# cloudctx v1.1 — hardening + lifecycle, from the 2026-07-06 code review

**Status:** approved design
**Date:** 2026-07-06
**Source:** full-code review of the CLI, shims, and tests (this session)

## 1. Objective

Close the review's defects and top missing features without changing the
architecture. Everything is additive; the env-isolation core is untouched.

## 2. In scope

### Defects

1. **Guard misses compound commands** (`shell/ctx.zsh`, `shell/ctx.bash`):
   `_cctx_guard` normalizes the line by replacing `&&`, `||`, `;`, `|`, `&`
   with newlines and runs the existing first-word check on **every** segment.
   The wrapper-skip loop also skips `-*` words so `stdbuf -oL az` is caught.
   Known accepted limitation: separators inside quoted strings cause segment
   splits (advisory-only guard; false positives are near-impossible in
   practice because the next word must be exactly `az`/`aws`).
2. **Non-string registry values crash** (`cloudctx` `load_registry`): values
   that are not `str` (tomllib bools/ints/dates from a hand-edited file) are
   dropped with a warning, never coerced — `azure_subscription = true` must
   not become the literal subscription "True".
3. **`install --write` not idempotent**: if the rc file already contains the
   exact source line, print "already installed" and exit 0 without appending.
4. **`status` can hang**: identity probes go through a `_probe_json(argv,
   timeout=10, env=None)` helper returning `(status, data)` where status is
   `ok | error | timeout`; a timeout prints `(timed out)` instead of hanging.

### Features

5. **`ctx delete <name>`**: removes the registry entry and the credential
   store `~/.cloudctx/<name>/`. Interactive `[y/N]` confirmation (EOF = no);
   `--force` skips it; `--keep-store` deletes only the registry entry.
   Regenerates iTerm2 profiles. Warns (does not block) when the context is
   active in the current shell. Deleting is irreversible — tokens are wiped.
6. **`--version`**: `__version__ = "1.1.0"` constant + argparse version flag.
7. **`ctx list -v`**: aligned columns — active marker, name, display, azure
   subscription (or tenant), aws profile. Plain `list` output unchanged.
8. **`ctx show <name>`**: prints the registry entry plus resolved store paths
   and whether each store directory/file exists.
9. **Internal `_names`**: prints bare context names, one per line (completion
   backend; also handy for scripting).
10. **Tab completion**: zsh `compdef` (registered only if `compdef` exists,
    i.e. compinit already ran) and bash `complete -F`. First arg completes
    subcommands; second arg completes context names via `cloudctx _names`
    for `use login exec open claude delete show`.
11. **Post-login verification** (`cmd_login`): after a successful non-dry-run
    Azure login, probe `az account show` in the context env and print the
    landed subscription/tenant. If the registry `azure_tenant` is GUID-shaped
    and differs from the landed `tenantId`, print a loud warning (domain-form
    tenants are not compared — false-positive risk).

## 3. Out of scope

- AWS SSO login automation (separate project, unchanged roadmap note).
- `ctx rename` (defer until a real need appears).
- Installer script / packaging (separate discussion).
- Landing-page updates beyond the README.

## 4. Compatibility and constraints

- Python 3.9+, stdlib only; both shims stay POSIX-ish and self-contained.
- All existing tests must keep passing unmodified; every change lands with
  its own regression test in `tests/test_cloudctx.py` / `tests/test_shell.py`.
- `_env` output contract unchanged (isolation core untouched).

## 5. Verification

`python3 -m unittest discover -s tests -v` green, plus
`sh tests/isolation_proof.sh` still passes.
